# MozaiksAI Runtime Integration (Control Plane)

MozaiksCore is a **control plane** and **session broker** for customer apps.

- MozaiksCore owns **authentication integration**, **authorization**, and **entitlements**.
- MozaiksAI is an **execution runtime** (workflows + ChatUI) and must not contain product logic.
- MozaiksCore does **not** reimplement ChatUI. It launches workflows into the runtime’s ChatUI.

This doc describes the integration points implemented in this repository.

## Capability Registry (Control Plane Owned)

Capabilities are defined locally in:

- `backend/core/config/ai_capabilities.json`
- Declarative spec files under `backend/core/config/capability_specs/*.json` (optional, app-owned)

Each capability maps to a runtime workflow ID:

```json
{
  "id": "ai_chat",
  "display_name": "AI Chat",
  "workflow_id": "chat",
  "allowed_plans": ["premium"],
  "requires_superadmin": false
}
```

Clients request **capabilities**, never workflow IDs.

### Declarative Capability Specs (Generated Upstream)

Apps can add capabilities via declarative spec files (typically generated by AppGenerator at app creation time).

MozaiksCore loads specs from:

- `MOZAIKS_AI_CAPABILITY_SPECS_DIR` (when set), otherwise `backend/core/config/capability_specs`

Each spec file is a single capability object (or `{ "capability": { ... } }`):

```json
{
  "id": "ask_mozaiks",
  "display_name": "AskMozaiks",
  "description": "Ask questions about this app.",
  "icon": "help-circle",
  "visibility": "user",
  "workflow_id": "chat",
  "enabled": true,
  "spec": {
    "system_prompt_template": "You are AskMozaiks for {{app_name}}...",
    "allowed_tools": ["search", "retrieve"],
    "retrieval_sources": [{ "type": "docs", "path": "docs/" }]
  }
}
```

MozaiksCore does not interpret `spec` fields; it treats them as runtime execution parameters.

## Backend API

All endpoints require `get_current_user()` authentication.

- `GET /api/ai/capabilities`
  - Returns the subset of capabilities the current user is entitled to run.
- `POST /api/ai/launch`
  - Input: `{ "capability_id": "..." }`
  - Output: `{ app_id, capability_id, chat_id, launch_token, expires_in, runtime: { runtime_ui_base_url, chatui_url_template, ... } }`

MozaiksCore does not execute workflows; it only returns metadata needed to open the runtime ChatUI.

The signed `launch_token` contains the server-constructed ExecutionContext (including `workflow_id` mapping and any `spec` execution parameters).

## Frontend UI

- `GET /ai` renders a simple capability launcher (`src/ai/AICapabilitiesPage.jsx`).
- Clicking **Launch** calls `POST /api/ai/launch`, then opens the runtime ChatUI URL in a new tab.

## Environment Variables

Required to launch workflows:

- `RUNTIME_BASE_URL` (runtime API base; also used as UI base by default)

Optional:

- `MOZAIKS_AI_CAPABILITY_SPECS_DIR` (where declarative capability specs live)
- `MOZAIKS_RUNTIME_UI_BASE_URL` (if ChatUI is hosted at a different origin than the runtime API)
- `MOZAIKS_CHATUI_URL_TEMPLATE` (how to build the ChatUI URL)

Execution token (issued by MozaiksCore in `POST /api/ai/launch`, validated by MozaiksAI):

- `MOZAIKS_EXECUTION_TOKEN_ALGORITHM` (default: `HS256`)
- `MOZAIKS_EXECUTION_TOKEN_EXPIRE_MINUTES` (default: `10`)
- `MOZAIKS_EXECUTION_TOKEN_ISSUER`
- `MOZAIKS_EXECUTION_TOKEN_AUDIENCE`
- `MOZAIKS_EXECUTION_TOKEN_SECRET` (HS* algorithms; defaults to `JWT_SECRET` when unset)
- `MOZAIKS_EXECUTION_TOKEN_PRIVATE_KEY` / `MOZAIKS_EXECUTION_TOKEN_PRIVATE_KEY_PATH` (RS*/ES* algorithms)

Template placeholders supported by the backend and frontend:

- `{runtime_ui_base_url} {app_id} {capability_id} {chat_id} {token}`

Example:

```env
RUNTIME_BASE_URL=http://localhost:8090
MOZAIKS_RUNTIME_UI_BASE_URL=http://localhost:8090
MOZAIKS_CHATUI_URL_TEMPLATE={runtime_ui_base_url}/chat?app_id={app_id}&chat_id={chat_id}&token={token}
```
